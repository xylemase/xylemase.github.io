1.1 What Is This Document?

This is a guide to Developer Documentation actions, and knowledge that should influence those actions, at Salesforce. The intent of this doc is to contain just the knowledge that you and I need as DevDoc writers, to speed us to it, and make it clear, down to the keystroke. Better still, it should allow you to copy commands and paste them whenever possible.

This guide arises from notes that I make as a reference primarily for myself, so that I can impose a structure, and re-find a particular link, or set of links among the disparate data sources at Salesforce, and convert them to keystrokes in writing.

This doc was originally intended to contain only notes on things that I had not found specifically mentioned. When I discover a body of relevant documentation, I provide the link to the existing doc. Ideally, maintaining this document will include recision of subject matter found to be covered better elsewhere as I discover it.

I’ve also found that some explanations exist but are are at a high level, and do not contain the specifics required to rapidly launch into keystrokes. Also, some reference data seems to be buried. I cover specific steps that get me to the goal and I surface the data that I’ve repeatedly needed.

Finally, a lot of this data is added to this document quickly, to capture it before I move on to execution. The level of finish and editing may not be equal to more widely known documents, and may include some idiosyncratic data. I am trying to separate out the parts of interest to others.

1.2 How Do I Use This Document?

When you have a new question, make a habit of going to the top of the document, looking the table of contents for the category that you need, and clicking through the successive tables of contents to narrow in on your desired knowledge. Get accustomed to using the browser "back" button to backtrack.

If you cannot find what you want through the categorizations of the TOC(s), go to the bottom of the document and consult the index. It is more like a tag, or keyword oriented approach, and offers alternatives to the categorizations.

If you still cannot find what you need, you can try doing a page or document search for the subject, phrase or related phrase. If it is a bona fide dev-doc related subject, and you have to do this, you should consider contacting me (mgipson@salesforce.com, or on Slack @Marty Gipson ), to request that I can include it in the document and make an index entry for it.

1.3 Why Aren’t the Contents of this Document in Confluence(/Quip/Google Doc/Slack)?

Confluence, Quip, Google Doc, Slack, and the subject matter they cover, are vast, and require a lot of indirection (jumping around) between documents, sections, and formats.

I needed one place in which to store at least a header for the subject matter, links to external documents, relationships between subjects, and alternatives between different ways to do the same thing.

To do this, I use a tool called Racket/Scribble. This tool automatically builds tables of contents (TOC), a robust index, internal links, external links, images and a clickable index from headers and index annotations in the text.

2.1 From Lint

Lint is a format checker that runs automatically while editing in VS Code, and outputs its problems to the problems tab. This is usually in the bottom-right part of the VS Code window.

Code Fences Require a Code Designator:

  • A ’code fence’ is a construct of three backtics ‘‘‘ in succession on its own line, both before and(λ (in) ) after a code example. It causes the code so enclosed to be rendered in a monospace font, usually Courier, or similar font, set off in a box. The opening code fence construct ought to have a language specifier following it immediately without spaces. A list of valid language specifiers can be found in in this Github listing.

  • One issue with this rule, that seems to require regularly breaking it, is that AMPscript does not have a language specifier. In order to avoid breaking the rule, I have been assigning "js" the short form of the javascript language specifier, whenever I encounter AMPScript.

2.7 From My Own Investigation

  • Proper Nouns (therefore capitalized):

    • Markdown: Markdown

    • PlantUML: PlantUML

  • Naming Conventions for Markdown Files

    • File names should be descriptive in a short string.

    • File names must be unqiue, at least within the repo. This can be checked by doing a shell find -name <filename> from the top of the tree

    • File names may need to be unique even more broadly than one’s repo, but in the answer linked to in the last item, Casey Smith did not know and suggested it as a question for SFDocs team. Also my intuition is that they must be unique so that any file directory mapping does not get confused. This suggests that attempting to put the file base name in as a URL on the document portal should work to find duplicate names in a broader context than one’s own accessible repos.

4.2 I18N and L10N in Salesforce Dev Docs

Internationalization (I18N), and localization (L10N).

The I18N Team has published a localization help doc.

Open API specs (OAS) can be localized. RAML specs can’t be localized. If your team is deciding whether to use RAML or Open API, let the product owner know that RAML can’t be localized.

The L10N team has said that RAML is not a format that they support, as it does not allow them to automate L10N in a way that is technically scalable. Common translation management software does not contain parsers that can protect the various formatters and differentiate between the translatable and non-translatable code snippets contained in the raml files. As a result, code will be corrupted, which will cause build and functionality failures.

4.3.1 Audit Trail API and Time Zones

Audit Trail stores data in the Central Standard Time (GMT-6, CST) time zone. Data pulled using the REST API has a CST timestamp.

According to Eddie Wuerch, the CST timezone is used becuase, "This is rooted in an ExactTarget decision made in the early 2000’s. ExactTarget started in Indianapolis, in the state of Indiana. At the time (this has since changed), most of Indiana, including Indianapolis, did not observe the summer time change, although much of the rest of the U.S. did. It’s something to do with data that had been recorded in the Central Time Zone, but ET and its servers were in Indiana with no Daylight Saving, so force all servers into CST and disable Daylight Saving Time. It was before my time (I started in 2006) and I never got the full story, and nothing I heard ever made sense. There won’t be a good explanation for this."

Also from Eddie Wuerch "I’m working on the response to a Sev.3 incident with Audit Trail ( https://gus.lightning.force.com/lightning/r/ADM_Work__c/a07EE00000s2rrVYAQ/view)

The customer reports a 6-hour delay in audit events. The data is not delayed - it’s being sent to them in ’MC server time’ (US Central Standard / no Daylight Saving), and is not being adjusted to their local time zone. CST is UTC-6, providing the 6-hour difference in times.

The customer is retrieving their audit event data by calling the Audit Trail REST API. Walking through the code, there appears to be nothing that considers time zone; data is directly output as stored in the database. The other option for customers to collect their Audit Trail events is to create an Audit Trail Extract Activity, which provides the option to adjust reported times to the customer’s time zone. That conversion occurs in the Extract code.

For the Audit Trail API, is it expected behavior for the times to be provided in CST? I couldn’t find anything in the docs.

-Eddie"

As a result of the email about that incident, Saikat Saha had responded, "Currently all audit events are stored in US CST as server timezone. Most customers (more than 97%) pull audit data using Automation Studio Data Extract activity which presents an option to convert the time in users local time, hence it is not an issue. Very few customers use REST API to pull audit data. REST API does not allow any time conversion. Supporting time conversion will take gigantic effort.

Proposing that we add a note in our user facing documentation and make it clear that the data is stored in our system in CST, audit data pulled using Audit Trail REST API has a timestamp in CST. Can you please add a note in the Audit Trail help documentation main page as well as audit trail REST api page?"

I am now (22 March, 2022) making this change to the Dev Doc.

Also, at the time this is being written, the U.S. Senate has unanimously passed a bill requiring all of the U.S. to move to permanent Daylight Savings Time/Summer Time Keeping. It still has to pass the House. I don’t think this will affect the records in place, but in may be interesting to see what happens in Indiana.

Also, I know that members on the Indiana team observe Eastern Time (EST, GMT-5). I don’t know how this squares with some of the statements above, but I will eventually ask about it, and update these notes.

4.4.1 Respect Threads

When someone makes a comment (i.e. v. "slacks") at the top level of a channel, it establishes a thread. Unless you mean to start a new thread in that channel (a comment that stands on its own), you should comment within the thread. Mouse over the comment and in the context menu, choose the "speech bubble" icon that has the tool-tip "Reply to thread", and make your comment there.

If you make a mistake and comment outside the thread (a common mistake), mitigate the error by the following steps:

  1. Select your out-of-thread comment.

  2. Copy it to the clipboard (command + c).

  3. Mouse over the top-of-thread Slack message and click the "Reply to thread" icon.

  4. Paste your text into the editor (command + v).

  5. Delete your out-of-thread comment.

This keeps your comment together with the original one, so that when someone selects that thread, they see your comment along with any others in the right-hand panel of the Slack UI.

6.4.4 The grep command

Grep is a command-line utility for searching files for lines that match a string, or a regular expression. Its name comes from the ed command g/re/p (globally search for a regular expression and print matching lines).

Many writers use utilities within their main editor to do a search. One of the issues that arises with the migration to a new toolchain is that a prior editor is not part of the new toolchain. I prefer to use grep and related commands, such as line tools because they work independently of the toolchain, they do not require the overhead of opening a main editor, and they

Grep has lots of different options; I only mention the ones of immediate use here. For a complete list of command-line options, you can execute the man page: "man grep".

Grep, as I usually use it, takes the basic form of

grep -Hi <text_to_search> <file_or_files_to_search>

Flags can be grouped together into one token, here seen as -Hi.

The "H" among the flags tells grep to print the pathname of the file containing any found line. This is necessary, especially in a recursive search, to tell us the path to the file we use.

The "i" tells it to do a case-insensitive match, applying the text represented by <text_to_search> without regard to case.

There is a flag "r" that you will discover if you use man grep. We don’t usually use this. When used with a directory name in place of <file_or_files_to_search> the "r" flag causes grep to descend recursively into that directory. It searches all files in that directory and contiues to descend in to all subordinate files. This is not very useful to us, because just blasts back every single instance of matching text in every single file found, many of which, we don’t want. We need finer control over the files and directories that grep looks at. We get that control by using the find command. I describe this further in the The bash/zsh find Command section

When grep is used as part of the find command, you will supply a placeholder that is substituted with each filename. This placeholder is a pair of curly brackets. Each curly bracket is escaped (with backslashes) to bypass special handling by the shell. The whole grep portion of the command is terminated with a semicolon, which is also escaped (backslashed)

The grep command in the context of a find command looks like this:

<body_of_the_find_command_described_next> -exec grep -Hi "text to find" \{\} \;

6.5.2 Making Reviewable Merges

To manage a large merge, plan to make series of smaller changes with the goal of a large merge, and make an intermediate branch to represent the large merge. I will describe this as an intermediate merge container branch, and will call it just the container branch.

Creating the container branch doesn’t have to be the very first thing you do with regard to the large merge. It may not have occurred to you right away that you want a large merge. So, a branch for a small merge can pre-exist the container branch. The container branch can be created even after creating the small merge branch, or branches.

But, you will want to have access to the container branch early, so that you can create the review requests as you finish them, and allow time for them to be reviewed while you finish the next change.

Create the container branch just as you would create a normal local branch:

Name the container branch something that reflects the common theme of the large merge. This could be a deadline date, a release number, or some other common theme, such as "HTAVs". The point of this branch is that it will provide a destination for each smaller change, and hold all the changes out of the main branch.

Be sure to push the branch to create a remote. You must do this to allow the name of the container branch to appear as a destination in place of "main".

Then, make smaller changes, each in their own branch, sized to be reviewable by one person. The name of these branches will usually incorporate a WI number. Make your changes in the branch. For each change, add the change, commit it, and push it. This will create a pull request notice on the repo’s GitHub page. This pull request notice’s destination will default to "main". You should be able to change the destination from "main" to the name of your container branch.

Merge each of these to the intermediate branch described in the last paragraph.

Frequently, a draft of documentation will come to you with code or response examples that are copied and pasted directly from a dev or QA machine. Or, you may notice data examples that contain specific information in pre-existing pages in your repos. There is a risk that these contain actual key values or other designators that should not be exposed to the wider world. There doesn’t seem to be any standard technique for anonymizing (changing) such data.

There should be a set of standards to guide each case toward a consistent style. The goals are to:

  • Remove any actual values for sensitive data such as keys, paths, object names, customer data.

  • Remove any basis for inferring or determining the actual values so removed. Don’t just change the number on a street address. Don’t make consistent use of rotation by an unvarying increment, such as rot1 or rot13), for example.

  • Retain some basis by which the user can infer a general type of data, such a GUID, or a string, such as by the formatting, or by the character domain (hex vs. base64 vs. readable string), or by the case that was originally used for hex-specific digits (a-f vs. A-F).

  • Retain data that conforms to, and supports the example being illustrated, such as pagination and count.

  • Clean up and obscure signs of internal process, organizational structure, actual people, and notes in such things such as object names and descriptions.

  • Where such anonymization has been completed, comment your markdown code with a "<!– Anonymization Done –>" to tell future maintainers (such as yourself) that you’ve completed the anonymization (and so, do not need to continue to re-anonymize).

Don’t leave original values for data that includes "key", or "user". Strongly consider altering values for any data that includes the words "id" or "name".

Alter values that are formatted as a globally unique identifier (GUID), such as "faf20007-9f6a-40f0-a990-9a7f0af9090f"

7.1.2 Avoid Providing a Basis for Inferring or Determining Sensitive Data

Don’t use the approach of consistently "rotating" values up or down by 1 or any other value such as 13 (half the American alphabet). These can be determined fairly easily and a correcting rotation will reveal the data. Don’t consistently just change every first (or last) character. If you have used any such practices consistently, an attacker can undo it.

7.1.3 Retain a Basis for Users to Infer General Type of Data

Don’t change prescribed values that are part of the domain of choices for that datum. Examples generally include "types" such as "locationType" or "authType".

If a value is a hex value, give some preference to "A" or "F" to substitute for alpha characters, and 0 or 9 to subtitute for numeric characters, with minor variations. Using these values a lot for hex makes it very obvious that the value is a hex value, and avoids the possibility of mapping values back to the actual value. Generally retain the alpha or number trait of characters that you replace with these, as well as the upper or lower case.

If a value is a base64 or base62 value, do your best to match, or at least allude to the frequency of placement of the wider-ranging characters. In other words, don’t alter a Base64 encoded string to have only hex digits 0-F, or even a higher percentage of them. Similarly, don’t replace a string that carries some meaning, such as a name, with a hex string, random alphanumeric numbers approximating base 62 or 64, or a binary string.

If a string’s value does not include spaces, and instead uses something like no spaces camel case, do your best to alter the actual contents, but reflect the same style.

If the particular distinction of base (62 or 64) matters, then state that it matters in the accompanying explanatory text. Don’t use characters counterinidicated by the stated base, such as slash ("/") or plus ("+") in base 62.

Use your best judgment as to when, whether, and how much to change URL’s. Don’t change URL’s that are actual examples. Do strongly consider changing URLs that exist among data when it is not a specific example of an endpoint. If you are altering a URL, or URL parameters, don’t change question marks ("?"), fragement (hash) marks ("#"), or url param delimiters ("&"), especially when they provide an example of structure that might be necessary for the case.

For the special class of hex strings that are globally unique identifiers, retain the dashed groups of hex digits and the dashes, but alter the values as described above.

Make a draft of your anonymized examples available to the PM and devs.

7.1.4 Retain Data that Supports and Conforms To An Example

If pagination is being illustrated by code example, don’t change things like page, pageSize, and count, nor the construction of the json data that it must agree with in the example.

7.1.5 Clean Up Signs of Internal Process, Organizational Structure, and People

Scan the examples for things such as "(Don’t Delete)", names of people, departments, or descriptions that are too revealing of actual internal process or the state of the object at the time that the sample was taken. Supplant these with other values, or make them vague.

7.3.1 XML

The following is a partial description. For a fuller description, see this wikipedia article on XML. Also, see the XML standard: IETF RFC 7303

XML consists of text elements bracketed between a start-tag, and an end-tag, or, an empty tag.

XML start-tags contain elements names which can be any text, as opposed to a predetermined set of element names, such as is used in HTML. The element name is bracketed by an opening angle/less-than symbol ("<"), and a closing angle/greater-than (">") symbol. The element name always sits just inside the opening angle bracket, and may or may not be followed by metadata, also called attributes. The attributes are name-value pairs: lvalues with an assignment symbol as an equals sign ("="), and a value being assigned. The value is typically a string, bracketed by double or single quotes, or a number, which is not bracketed by quotes at all.

XML end-tags consist of a opening-angle/less-than symbol followed immediately by a forward slash, ("</"), then an element name which matches the start-tag element name, then a a closing angle/greater-than (">") symbol.

An XML empty tag does not have a closing tag, and indicates that it is the sole tag by including the forward slash at the end, as in <line-break />. I am not sure whether there needs to be a space between the element name and the forward slash, but it is shown that way in an example in the Wikipedia page cited above.

XML is the format used in the legacy toolchain, associated with Perforce (P4), and Oxygen (O2).

7.6.1 VIM Search

To enter search mode from command mode, type a single forward slash ("/"). Remember that the search string is a regular expression. Also remember that numerous regular expression tokens need to be escaped in Vim

In Vim, you can do a negative lookahead. I think this is done by grouping the part of the pattern that you want NOT to match, and following the group with @!. Here is an example of a negative lookahead I did on the information topology JSON output to find all instances of a color that is NOT the default color.

^\s\+"color": \("#ffffff"\)\@!

This found every single instance of such colors, both in Source and Targets. Since Source attributes appeared consistently leftward of Targets at an 8 space indent (on a json.dump(), I could limit the search to only Source instances by quantifying the leading space count as exactly 8:

^\s\{8}"color": \("#ffffff"\)\@!

7.6.2 VSCode Search

In VSCode with Vim key bindings (the way I use VSCode), you enter search mode the same way as you do in Vim. From command mode (after hitting Esc), you just type a slash.

The search expression is a regular expression (regex), but it usually differs from the syntax used in Vim, in that you rarely need to escape the tokens that you need to escape in Vim.

9.3.5 Git Stash a Branch’s Changes

I am still trying to form the mental context for the Salesforce-specific circumstances surrounding the need to do a stash. A StackOverflow article recommends against relying on stash, advising the user to prefer frequent commits instead.

I am not sure that this advice works for everyone. I think, at Salesforce, especially, with the way that releases work from SF Docs there may be an increased need for stashing to preserve work that has a specific start date in production. In other words, you don’t want to mistakenly release some changes early, and you need to stash work to hide the changes from an inadvertent merge, but commit and merge them later in a short window of time just before the release.

You could just "stash" changed in a directory just outside of Git management. I have been following the policy of doing that, as insurance. But, Git offers the stash functionality, and one should learn to use it correctly.

A few things I have discovered through experience:

  • Creating and switching to a branch, changing a file in that branch, then switching back to main, doing a pull in main, then starting a new branch, does NOT restore the changed file. If you want to keep a copy of the changed file but restore the status quo ante change, you must somehow stash the file.

  • Stashing with changed but un-(GIT)-tracked (not yet added) file loses those changes in the stash. In other words, when you restore the stash, if you had not added the file, it WILL be lost to you. There is a flag, --include-untracked, that is supposed to stash the un-added files, but this raises some file management issues itself (saving .swp files and so on). There are some actions that can mitigate this loss:

    • Add everything that you will need access to when you restore, doing so before any stash. If you can do this, it is probably the best solution.

    • As insurance, create a directory outside of Git management, and hand copy all your future-needed files there.

    • If you must refrain from adding all files, use the --include-untracked flag with your git stash comand.

    • If, somehow, you lose a file, there is a last ditch measure. See if the file remains open the VSCode buffer. VS Code may not permit saving it, but you can always open an alternate editor, such as gVim, copy from the VSCode buffer, to paste in the alternate editor. Save it into the directory it had originally occupied, and go on from that point as normal.

To stash, be sure you are in the branch you want to stash. Ensure that you consider each of the items in the last point.

Issue the command:

git stash

You should see a positive response of some kind. This is what I saw in one case:

mgipson@mgipson-ltmx7ad marketing-cloud-docs % git stash

Saved working directory and index state WIP on JWT_Ampscript_Commands: aac0302 Merge pull request #85 from salesforcedocs/W-10432780_Add_detail_to_Java_SDK_Page_getting-started-with-the-java-sdk.md

If you want to see the files that were included in the stash just made, you can issue the command

git stash show stash@{0}

At this point, and after having seen your files in the last command, you have your changes "git stashed".

One likely path now is to move on to a branch that does not contain the changes or any new files you have stashed.

To do this "git switch" back to main, then follow the instructions in Clean Your Working Directory to a Pristine Match with "main" (at the time this is written, it happens to be the next section). You may need to run through it a couple of times or more until your main branch seems really clean. Keep testing by doing a git status and/or git pull until it says that it is "up to date with main".

At this point, you may want to let your stash persist, holding your files until you need them. Or, you may want to create a new branch and immediately pull your files into it. This is the likely case when, say, you realize you want to have changes from several separate prior branches in one branch that is destined for a release.

When you want the stashed files back, make sure you are in your existing desired branch with:

git switch <your desired existing branch name>

then, pop the branch off the stash-stack like this:

git stash pop

The pop command will, by default, take the 0th stashed set of files off the stack.

9.3.9 Gitignore a file

Some activities among Git-managed files may result in new files being created that are not part of Git-managed files. Such things as redirecting output of a grep to a file, copying a file for later comparison, are valid things to do, which result in a new file. Obviously, these files should not be committed.

There are ways to mark a file as "ignored", that is, outside of management by Git. One is to add the file’s name to a .gitignore file. Although one external link where this is described is an article for Atlassian BitBucket, it has to do with Git, is very clear. Reiterated here:

  • Git sees every file in your working copy as one of three things:

    1. Tracked - a file which has been previously staged or committed;

    2. Untracked - a file which has not been staged or committed;

      or

    3. Ignored - a file which Git has been explicitly told to ignore.

  • Ignored files are tracked in a special file that is:

    • Named ".gitignore"

    • Located in the root of your Git-managed files.

    • Expressed in .gitignore either explicitly or as a pattern.

After you create a .gitignore file in your repo, and set patterns in it to match files that you don’t want Git to track. A pattern that I usually add to my .gitignore is:

# ignore gvim/mvim swap files

*.sw?

Git will start tracking repository files and respect the patterns set in the .gitignore file after you run the git add command.

But if you make later changes to the .gitignore file to allow tracking a file that you previously set to be ignored, the file will still be untracked if you only run the

git add .

command. The final step needed is to clear the git cache. This can be done with:

git rm -r --cached .

Then, run the

git add .

command.

To check whether a previously tracked type of file is now ignored, run the

git status

command.

It seems to me, from having tried it, that adding patterns to the .gitignore is picked up as soon as the .gitignore undergoes a git add, without necessarily clearing the cache.

For more about this, see this external article about .gitignore.

One question I still have is whether such changes to the .gitignore file should be done in just any branch, or in main, or in a special branch. The answer is not in the page cited above. Thinking about it, I think doing it in main will never be the right way to go, and it could be either a special branch, or just any branch that you happen to be in. As long as that branch will eventually be merged, it will be the correct way.

9.4.2 What If I Started Changing Files Before Creating a New Branch?

So, let’s say that you finished your changes, merged a branch, deleted the branch (so you’re in main. Then, you got some news about some changes that need to be made and you forgot to make a new branch for them. You just eagerly went ahead and started making the changes. The changes are sitting in your VS Code editor, with a big white dot next to the title (indicating unsaved changes). You want to make a new branch but are not sure whether this will mess anything up by overriding the contents of the file, or anything else.

The answer is that making a new branch won’t mess anything up. You can keep the changed text in the VS Code buffer, but make the new branch on the command line. Then, as long as you are in the branch (which should be the outcome of making the new branch with a git switch -C <new_branch_name>), the VS Code editor will "occupy" the new branch, and you can just save the changed file.

9.4.4 Quickly Convert A Quip API Draft to a Markdown Draft

Use VSCode to Mitigate Common Quip Export Header Faults

Quip won’t let you select the parts of the page that you want to export. The way I usually handle this is to paste the entire page’s exported Markdown into a single file, then section up the parts to go into different files.

Prepare the Markdown file, usually by copying the whole file from the PM’s copy, to your own copy in your private space. Use your copy to make modifications. If there are columns showing "Required", fold their contents into the Description, then delete the whole column.

Prepare VSCode to receive Markdown by creating a file, named to place it among the files you are editing. Call the file something like <name_first_part>_md_landing_&_worksheet where "<name_first_part>" matches the first part of the name of other files you are adding.

In Quip, with the desired page open, from the "Document" menu, click on the "Export" command. From the submenu, choose "Markdown". See the confirming dialog:

Document copied to clipboard as Markdown

In the VSCode buffer, open on the "<name_first_part>_md_landing_&_worksheet" file, paste the copied Markdown text.

From here, you can section the combined document up into the method pages.

Use VSCode to Mitigate Common Quip Export Header Faults

Quip, or the way some PM’s use Quip in their API drafts, results in bolded headers, with double asterisks on either side of the header text. Also, headers may undergo a sort of header demotion, where headers that are typically H2 become H3 or lower.

Below are a search, and search and replace expression using that same search, to put the most common H2 headers into their most usual or "normal" form.

VSCode, with Vim bindings, supports the same search and replace that GVim provides. VSCode does not require the same escaping that GVim requires, so the GVim equivalent expressions are described in their own section, Use GVim to Mitigate Common Quip Export Header Faults

9.6.1 How to Undo Light Gray Text Color in Google Docs Text Pasted From a VSCode Preview

One of the easiest ways to create a dev doc draft in Google Docs for editor review is to paste the text from a VSCode SFDocs preview. But this text always seems to paste in a light gray color. The editors object to this, and cannot use it.

  3. Ensure that you have opened the SFDocs preview on the doc for which you need a draft. Also ensure that you have open a new, blank, Google doc.

  5. Copy it to the system clipboard.

  6. In the new, blank Google doc, paste the preview text.

  8. In the Google doc, re-select the whole set of pasted text.

  9. Click the toolbar "Clear Formatting" icon, which looks like a capital "T" with an upper-left to lower-right slash through it. Alternatively, use the key combination Command + backslash.
